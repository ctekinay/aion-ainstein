# Developer Specification: `interactive-preview` Plugin for AInstein

**Version:** 0.2.0-draft  
**Date:** 2026-05-05  
**Author:** Laurent (requirements) + the assistant (spec)  
**Status:** Draft — template under validation in AInstein

---

## 1. Intent & Vision

### 1.1 Problem Statement

When working with AInstein in the terminal, there is no equivalent of the an inline rendered-artifact experience — the ability to instantly see a rendered, interactive visualization in the browser. Developers must manually copy generated code, set up a project, and open a browser. This breaks the creative loop.

### 1.2 Goal

Deliver a AInstein plugin that provides an artifact-like experience: the assistant generates an interactive ArchiMate JSX visualization, and the developer instantly sees a rendered, clickable result in their browser — with the same quality as the best artifacts produced in AInstein.

### 1.3 Scope

The plugin produces **ArchiMate architecture viewers**. The template is a fixed, fully specified reference component. the assistant's only job is to populate the data structure — the rendering logic, symbols, relation styles, and layout conventions are embedded in the template and must not be modified by the assistant.

### 1.4 Development Approach

1. **Template-first:** The reference JSX template (`archimate-viewer-template.jsx`) is developed and validated in AInstein as a rendered artifact, iterated until the user is satisfied.
2. **Then plugin:** Once the template is approved, it is packaged into the AInstein plugin structure with the MCP server, skill, hook, and command.

│       └── vite.config.js           # Vite configuration

### 2.1 Component Responsibilities

| Component | Role |
|---|---|
| **Skill** (`interactive-view/SKILL.md`) | Instructs the assistant to parse ArchiMate XML and populate the VIEWS data structure. Contains the exact data format, element type mappings, and layout rules. Explicitly prohibits modifying template rendering logic. |
| **MCP Server** (`.mcp.json`) | Provides tools: `preview_start`, `preview_update`, `preview_screenshot`, `preview_stop`. Manages the Vite dev server lifecycle. Configurable: can persist between sessions or start/stop per session. |
| **Hook** (`post-file-write.sh`) | PostToolUse hook: detects when the assistant writes a `.jsx` file to the preview directory and auto-triggers the preview server. |
| **Command** (`preview.md`) | `/preview` — explicit trigger to start, restart, or stop the preview. |
| **Template** (`templates/archimate-viewer/`) | The fixed, validated reference component. the assistant injects data only — never modifies rendering logic, icons, relation styles, or layout. |

**Note:** No reviewer agent is needed. Quality is guaranteed by the fixed template — correct ArchiMate symbols, relations, and layout are baked into the rendering logic. the assistant cannot produce incorrect visuals as long as it uses valid element and relation types.

## 3. User Requirements — JSX Visualization

These requirements are derived from reviewing existing JSX artifacts, with the **bean-webshop ArchiMate viewer** as the gold standard, enhanced with UX features from the **NS Content Strategie viewer**.

### 3.1 ArchiMate Compliance (HARD REQUIREMENTS)

| ID | Requirement | Priority |
|---|---|---|
| **R-AM-01** | Element symbols MUST conform to ArchiMate 3.1 specification. Each element type has its own SVG icon rendered inside the element box (top-right corner). Icons are defined once in the template and never generated by the assistant. | Must |
| **R-AM-02** | Relation line styles MUST conform to ArchiMate 3.1: solid vs dashed patterns, and correct arrowhead/diamond shapes per relation type. Marker definitions are fixed in the template. | Must |
| **R-AM-03** | Layer colors MUST follow ArchiMate conventions: Motivation (purple), Strategy (tan/amber), Business (yellow), Application (blue), Technology (green). | Must |
| **R-AM-04** | Elements MUST be arranged in horizontal layer bands, with Motivation at top progressing down to Technology. | Must |
| **R-AM-05** | Line routing MUST minimize crossings. Use manhattan routing (orthogonal paths). For models with 50+ elements, integrate a layout algorithm (dagre or elkjs). | Must |

### 3.2 Interactivity (HARD REQUIREMENTS)

| ID | Requirement | Priority |
|---|---|---|
| **R-IX-01** | Click-to-select: clicking an element highlights it and its directly related elements. Non-related elements and relations are dimmed (opacity ≤ 0.12). | Must |
| **R-IX-02** | Side panel: when an element is selected, a detail panel appears showing: element type, name, documentation, layer badge, and a list of all relations with clickable navigation to related elements. | Must |
| **R-IX-03** | Zoom: scroll-wheel zoom with +/− buttons and a reset button. Controls positioned in a top toolbar. | Must |
| **R-IX-04** | Pan: click-and-drag on the canvas background to pan the view. | Must |
| **R-IX-05** | View switcher: when the model contains multiple views, provide toggle buttons in the top toolbar. Switching resets zoom/pan and selection. | Must |
| **R-IX-06** | Legend: always present, showing layer colors with labels. Position: bottom bar of canvas area. | Must |

### 3.3 SVG Export (DESIRED)

| ID | Requirement | Priority |
|---|---|---|
| **R-EX-01** | Provide a "↓ SVG" button in the top toolbar that exports the current visible state as a standalone SVG file. | Should |
| **R-EX-02** | The exported SVG MUST use SVG `<text>` elements (not `<foreignObject>`) to ensure compatibility with vector editors and browsers. This means the export function rebuilds text rendering for the export. | Should |
| **R-EX-03** | The exported SVG must be self-contained (inline styles, embedded markers, no external dependencies). | Should |

### 3.4 Visual Quality

| ID | Requirement | Priority |
|---|---|---|
| **R-VQ-01** | Font: `system-ui, -apple-system, sans-serif`. Element names 11px bold, type labels 9-10px. | Must |
| **R-VQ-02** | Element boxes: rounded corners (rx=4), subtle border (1.2px), fill color matching layer. | Must |
| **R-VQ-03** | Selected element: yellow highlight fill (#FFF9E6), red border (#E74C3C, 2.5px). Active relations turn red with matching markers. | Should |
| **R-VQ-04** | Dimmed state: opacity ≤ 0.12 for both elements and their relations. | Should |
| **R-VQ-05** | Side panel: white background, clear typography hierarchy, clickable relation cards with hover effect. | Must |
| **R-VQ-06** | Canvas background: light neutral (#F8F9FA). Top toolbar and bottom legend bar: white with subtle border. | Should |

### 3.5 Robustness

| ID | Requirement | Priority |
|---|---|---|
| **R-RO-01** | The JSX MUST parse and render without errors in the the AInstein preview sandbox AND in a local Vite dev server. | Must |
| **R-RO-02** | Only use React hooks from `react` (useState, useRef, useCallback). No external state management. | Must |
| **R-RO-03** | Only use libraries available in the the AInstein preview sandbox: React, lucide-react, recharts, d3, Tailwind (utility classes only). The template uses inline styles only (no Tailwind dependency). | Must |
| **R-RO-04** | The component MUST have a `export default` function component with no required props. | Must |
| **R-RO-05** | No use of `localStorage`, `sessionStorage`, or any browser storage APIs. All state in React state. | Must |

## 4. Template Architecture

### 4.1 Data-Driven Design

The template enforces a strict separation between **data** (generated by the assistant) and **rendering** (fixed in template). the assistant's only task is to produce the `VIEWS` object.

```
┌─────────────────────────────────────────────┐
│  VIEWS = {                                  │  ← the assistant generates this
│    "view-key": {                            │
│      label: "Human-readable Label",         │
│      elements: [                            │
│        { id, name, type, layer,             │
│          x, y, w, h, doc }                  │
│      ],                                     │
│      relations: [                           │
│        { from, to, type }                   │
│      ],                                     │
│    },                                       │
│    ...                                      │
│  }                                          │
├─────────────────────────────────────────────┤
│  LAYER colors          — fixed              │
│  REL_STYLES            — fixed              │
│  ArchiIcon()           — fixed              │
│  MarkerDefs()          — fixed              │
│  Manhattan routing     — fixed              │
│  View switcher         — fixed              │
│  Side panel            — fixed              │
│  Zoom/pan              — fixed              │
│  Highlight/dim         — fixed              │
│  SVG export            — fixed              │
│  Legend                 — fixed              │
└─────────────────────────────────────────────┘
```

### 4.2 VIEWS Data Format

```typescript
type ElementData = {
  id: string;           // Unique identifier (e.g., "ba1")
  name: string;         // Display name (e.g., "Laurent")
  type: ElementType;    // ArchiMate element type (e.g., "BusinessActor")
  layer: LayerKey;      // "motivation" | "strategy" | "business" | "application" | "technology"
  x: number;            // X position in canvas coordinates
  y: number;            // Y position in canvas coordinates
  w: number;            // Width (typically 120-220)
  h: number;            // Height (typically 44-60)
  doc: string;          // Documentation / description
};

type RelationData = {
  from: string;         // Source element ID
  to: string;           // Target element ID
  type: RelationType;   // ArchiMate relation type (e.g., "Serving")
};

type ViewData = {
  label: string;        // Human-readable view name
  elements: ElementData[];
  relations: RelationData[];
};

type Views = Record<string, ViewData>;
```

### 4.3 Supported Element Types (all rendered with fixed ArchiIcon)

**Motivation:** Stakeholder, Driver, Assessment, Goal, Outcome, Principle, Requirement, Constraint, Value, Meaning  
**Strategy:** Capability, Resource, CourseOfAction, ValueStream  
**Business:** BusinessActor, BusinessRole, BusinessService, BusinessProcess, BusinessObject, BusinessInterface, BusinessEvent, BusinessFunction, Contract, Product  
**Application:** ApplicationComponent, ApplicationService, ApplicationInterface, ApplicationFunction, DataObject  
**Technology:** Node, SystemSoftware, Artifact, CommunicationNetwork, TechnologyService, Device  

### 4.4 Supported Relation Types (all rendered with fixed REL_STYLES + MarkerDefs)

| Relation | Line | Source Marker | Target Marker |
|---|---|---|---|
| Association | solid | none | none |
| Serving | solid | none | open arrow |
| Composition | solid | filled diamond | none |
| Aggregation | solid | open diamond | none |
| Assignment | solid | none | filled arrow |
| Realization | dashed 6-3 | none | open arrow |
| Triggering | solid | none | filled arrow |
| Flow | dashed 8-4 | none | filled arrow |
| Access | dashed 4-3 | none | open arrow |
| Influence | dashed 6-3 | none | open arrow |
| Specialization | solid | none | open triangle |

## 5. MCP Server Specification

### 5.1 Technology

- **Runtime:** Node.js (TypeScript)
- **MCP SDK:** `@modelcontextprotocol/sdk`
- **Dev server:** Vite
- **Screenshot:** Puppeteer (headless Chrome)
- **Transport:** stdio
- **Lifecycle:** Configurable — can persist between sessions or start/stop per session.

### 5.2 Tools

| Tool | Input | Output | Description |
|---|---|---|---|
| `preview_start` | `{ code: string }` | `{ url: string, message: string }` | Scaffolds Vite project from template, injects the VIEWS data into the JSX, starts dev server, opens browser. Returns localhost URL. |
| `preview_update` | `{ code: string }` | `{ message: string }` | Hot-replaces the VIEWS data in the running project. Vite HMR handles browser refresh. |
| `preview_screenshot` | `{ selector?: string, fullPage?: boolean }` | `{ image: base64, width: number, height: number }` | Takes a screenshot via Puppeteer for the assistant to verify rendering. |
| `preview_stop` | `{}` | `{ message: string }` | Stops Vite dev server and cleans up temp files. |

### 5.3 Lifecycle

```
AInstein session starts
  └─ Plugin loads → MCP server available (tools registered)
  
User: "visualize this architecture"
  └─ the assistant reads SKILL.md
  └─ the assistant parses ArchiMate XML → generates VIEWS data
  └─ the assistant calls preview_start({ code: "const VIEWS = {...}" })
      └─ MCP: mkdir temp project, copy template files
      └─ MCP: inject VIEWS data into template
      └─ MCP: vite dev (cached node_modules)
      └─ MCP: open browser at localhost:5173
      └─ Returns { url: "http://localhost:5173" }
  └─ User views result in browser
  └─ If changes needed: the assistant calls preview_update with new VIEWS data
  
User: "/preview stop" or session ends
  └─ the assistant calls preview_stop()
```

## 6. Skill Specification (`interactive-view/SKILL.md`)

### 6.1 Trigger Conditions

The skill activates when the user mentions: ArchiMate, architecture diagram, architecture visualization, "show me the architecture", "visualize this model", or references an ArchiMate Open Exchange XML file.

### 6.2 Workflow

1. **Ensure data source** — require ArchiMate Open Exchange XML. Coordinate with archimate-oxc-generator and archimate-oxc-view-generator skills as needed.
2. **Parse XML** — extract elements (identifier, xsi:type, name, documentation), relations (source, target, xsi:type), and views.
3. **Map to VIEWS format** — convert parsed XML to the exact VIEWS data structure (see §4.2). Compute x/y/w/h layout positions following layer-band conventions.
4. **Call preview_start** — send the VIEWS data to the MCP server.
5. **Iterate** — if user requests changes, modify VIEWS data and call preview_update.
6. **Deliver** — provide the localhost URL. Optionally export the XML model as a downloadable file.

### 6.3 Key Instructions for the assistant (embedded in SKILL.md)

- The exact VIEWS data structure format with TypeScript-style type definitions
- The complete element type → layer mapping table
- The complete relation type list (the assistant must use exact type strings)
- Layout rules: layer ordering (motivation top → technology bottom), minimum element spacing, preferred element dimensions per type
- **Prohibition:** never modify the template rendering logic, ArchiIcon function, MarkerDefs, REL_STYLES, or LAYER constants — only populate the VIEWS data
- **Prohibition:** never add inline styles or custom rendering to individual elements

## 7. Hook & Command

### 7.1 Hook: `post-file-write`

- **Event:** PostToolUse (Write tool)
- **Condition:** file path matches `**/preview/**/*.jsx`
- **Action:** calls `preview_update` with the file contents

### 7.2 Command: `/preview`

```markdown
---
description: Start, update, or stop the interactive ArchiMate preview server
allowed_tools: ["mcp__preview__preview_start", "mcp__preview__preview_update", 
                "mcp__preview__preview_stop", "mcp__preview__preview_screenshot"]
---

Start or manage the interactive preview. Usage:
- `/preview` — start preview with the latest generated VIEWS data
- `/preview stop` — stop the preview server
- `/preview screenshot` — take a screenshot of the current state
```

## 9. Design Decisions (Resolved)

| # | Decision | Rationale |
|---|---|---|
| 1 | MCP server lifecycle is **configurable** (persist or per-session). | Different workflows need different behavior. |
| 2 | Template uses **inline styles only** (no Tailwind). | Maximum portability — works in the AInstein preview sandbox and local Vite without Tailwind config. |
| 3 | Plugin quality target: **publishable quality**, but kept private initially. | Build to public standards from the start to avoid tech debt. |
| 4 | Large models (50+ elements): integrate **dagre/elkjs** for auto-layout. | Manual x/y positioning doesn't scale. Layout algorithm handles element placement. |
| 5 | SVG export uses **SVG `<text>` elements** (not foreignObject). | foreignObject doesn't render in exported SVGs. Export function rebuilds text for compatibility. |
| 6 | No reviewer agent needed. | Template rendering is fixed — correct ArchiMate visuals are guaranteed by the template, not by post-hoc evaluation. |

## 10. Appendix: Reference Template

The reference template is `archimate-viewer-template.jsx`, currently being validated in AInstein. It combines:

- **Bean-webshop accuracy:** correct ArchiMate 3.1 symbols (ArchiIcon), correct relation styles (REL_STYLES + MarkerDefs), layer-band layout, legend
- **NS-viewer UX:** view-switcher, click-to-highlight with dimming of unrelated elements, side panel with relation navigation
- **SVG export:** placeholder button in top toolbar (implementation in Phase 1)
- **Robustness:** inline styles only, no external dependencies beyond React, `export default` function component, no browser storage APIs

### Reviewed Artifacts

| Artifact | Verdict |
|---|---|
| **Bean Webshop** viewer | **Gold standard.** Correct symbols, relations, layers. Good interactivity and readability. |
| **NS Content Strategie** viewer | Symbols/colors not ArchiMate-compliant, but excellent UX: view-switcher, highlight/dimming, side panel with relation navigation. |
| **Winter Events** (regenerated) | Less accurate symbols/relations than bean-webshop. Zoom controls poorly placed. No SVG export. |
| **Winter Events** (original) | Parsing failed — demonstrates need for robustness requirements (R-RO-01 through R-RO-05). |

### Target

The approved template merges bean-webshop accuracy + NS-viewer UX + SVG export, with guaranteed parse reliability.
